## Kigali City Services & Places Directory

## 1. Database Structure

### Firestore Collections

The application uses two main Firestore collections:

#### **users/** Collection
Stores user profile information for authenticated users.

**Document ID:** User's Firebase Auth UID

**Fields:**
- `uid` (String): User's unique identifier from Firebase Auth
- `email` (String): User's email address
- `emailVerified` (Boolean): Email verification status
- `notificationsEnabled` (Boolean): User's notification preference
- `createdAt` (Timestamp): Account creation date

**Purpose:** Maintains user profiles and preferences separate from Firebase Auth, allowing for additional user data and settings.

**Security:** Users can only read/write their own profile documents.

---

#### **listings/** Collection
Stores all service and place listings created by users.

**Document ID:** Auto-generated by Firestore

**Fields:**
- `name` (String): Name of the service or place
- `category` (String): Category type (Hospital, Police Station, Library, etc.)
- `address` (String): Physical address
- `contactNumber` (String): Phone number
- `description` (String): Detailed description
- `latitude` (Double): Geographic latitude coordinate
- `longitude` (Double): Geographic longitude coordinate
- `createdBy` (String): UID of the user who created the listing
- `timestamp` (Timestamp): Creation date/time

**Indexes Required:**
- Composite index on `createdBy` + `timestamp` (descending) for user listings query
- Single-field index on `timestamp` (descending) for all listings query

**Purpose:** Central repository for all location data, enabling real-time synchronization across all users.

**Security:** 
- Read: All authenticated users
- Create: Authenticated users (must set `createdBy` to their UID)
- Update/Delete: Only the creator of the listing

## 2. Data Model Design

### ServiceListing Model

The `ServiceListing` model encapsulates all information about a service or place:

```dart
class ServiceListing {
  final String? id;
  final String name;
  final String category;
  final String address;
  final String contactNumber;
  final String description;
  final double latitude;
  final double longitude;
  final String createdBy;
  final DateTime timestamp;
}
```

**Design Decisions:**
- `id` is nullable because it's only available after Firestore document creation
- Coordinates stored as `double` for precision
- `timestamp` stored as `DateTime` for easy manipulation in Dart
- `createdBy` links listing to user, enabling access control

**Validation:**
- All fields are required (non-nullable except id)
- Category must be one of predefined options
- Coordinates validated as valid latitude/longitude ranges
- Contact number and address validated for non-empty strings

---

### UserProfile Model

```dart
class UserProfile {
  final String uid;
  final String email;
  final bool emailVerified;
  final bool notificationsEnabled;
  final DateTime createdAt;
}
```

**Design Decisions:**
- Mirrors Firebase Auth user data for easy access
- `notificationsEnabled` provides user preference storage
- Separate from Auth to allow extension without Auth modifications

## 3. State Management Implementation

### Architecture Choice: Provider

**Rationale:**
- **Simplicity**: Easy to learn and implement
- **Integration**: Native Flutter support
- **Real-time**: Excellent compatibility with Firestore streams
- **Scalability**: Sufficient for current app complexity
- **Performance**: Efficient rebuilds with Consumer widgets

### Provider Structure

#### **AuthProvider**
Manages authentication state and user profile data.

**Responsibilities:**
- Listen to Firebase Auth state changes
- Execute sign up, sign in, sign out operations
- Load and cache user profile from Firestore
- Update user preferences
- Expose loading and error states

**Key Features:**
- Automatically loads user profile when user signs in
- Provides `isAuthenticated` computed property
- Handles email verification enforcement
- Caches user data to minimize Firestore reads

**State Properties:**
```dart
User? _user;
UserProfile? _userProfile;
bool _isLoading;
String? _errorMessage;
```

---

#### **ListingsProvider**
Manages listings data, search, and filtering.

**Responsibilities:**
- Listen to Firestore listings streams
- Execute CRUD operations (Create, Read, Update, Delete)
- Apply search and category filters
- Expose filtered listings to UI
- Manage loading and error states

**Key Features:**
- Real-time updates via Firestore streams
- Client-side filtering for instant search results
- Separate streams for all listings and user listings
- Optimistic UI updates with error handling

**State Properties:**
```dart
List<ServiceListing> _allListings;
List<ServiceListing> _userListings;
List<ServiceListing> _filteredListings;
String _searchQuery;
String _selectedCategory;
bool _isLoading;
String? _errorMessage;
```

**Filtering Logic:**
The provider applies filters client-side for instant updates:
```dart
void _applyFilters() {
  _filteredListings = _allListings.where((listing) {
    bool matchesSearch = _searchQuery.isEmpty ||
        listing.name.toLowerCase().contains(_searchQuery.toLowerCase());
    
    bool matchesCategory = _selectedCategory == 'All' ||
        listing.category == _selectedCategory;
    
    return matchesSearch && matchesCategory;
  }).toList();
  
  notifyListeners();
}
```

## 4. Service Layer Design

### AuthService
Handles all Firebase Authentication operations.

**Methods:**
- `signUp()`: Create account and send verification email
- `signIn()`: Authenticate with email verification check
- `signOut()`: Clear authentication state
- `sendVerificationEmail()`: Resend verification email
- `getUserProfile()`: Fetch user profile from Firestore
- `updateNotificationPreference()`: Update user settings

**Design Pattern:** Repository pattern
- Abstracts Firebase Auth complexity
- Provides consistent error handling
- Returns user-friendly error messages

---

### FirestoreService
Handles all Firestore operations for listings.

**Methods:**
- `getAllListings()`: Stream of all listings
- `getUserListings()`: Stream of user's listings
- `createListing()`: Add new listing
- `updateListing()`: Modify existing listing
- `deleteListing()`: Remove listing
- `getListing()`: Fetch single listing

**Design Pattern:** Repository pattern
- Encapsulates Firestore operations
- Provides type-safe data access
- Enables easy testing and mocking

## 6. Design Trade-offs

### Trade-off 1: Client-side vs Server-side Filtering

**Decision:** Client-side filtering

**Rationale:**
- Instant search results without network latency
- Reduced Firestore read operations
- Simpler implementation
- Acceptable dataset size (<1000 listings expected)

**Drawback:**
- Not suitable for very large datasets
- All listings loaded into memory

**Future Consideration:**
If dataset grows significantly, migrate to Firestore queries with indexes.

---

### Trade-off 2: Real-time vs On-demand Updates

**Decision:** Real-time streams

**Rationale:**
- Automatic UI updates
- No manual refresh needed
- Better user experience
- Showcase Firestore capabilities

**Drawback:**
- Continuous connection required
- Slightly higher bandwidth usage

**Mitigation:**
Only active screens maintain stream listeners.

---

### Trade-off 3: Email Verification Enforcement

**Decision:** Block access until verified

**Rationale:**
- Security best practice
- Prevents fake accounts
- Ensures valid contact information

**Drawback:**
- May frustrate users who forget to verify
- Additional step before app access

**Mitigation:**
Clear messaging and option to resend verification email.

## 7. Technical Challenges

### Challenge 1: Coordinate Input
**Problem:** Users need to input latitude/longitude coordinates, which is not user-friendly.

**Solution:** Provide default Kigali center coordinates (-1.9441, 30.0619) and allow users to adjust or use their current location.

**Future Enhancement:** Integrate location picker with map tap or address geocoding.

---

### Challenge 2: Map API Key Configuration
**Problem:** Google Maps requires API key setup, which varies by platform.

**Solution:** Clear documentation in README with step-by-step instructions. Placeholder value in code with clear comment.

---

### Challenge 3: Firestore Security
**Problem:** Ensuring users can only modify their own listings.

**Solution:** 
- Store `createdBy` UID in each listing
- Implement Firestore security rules to verify ownership
- UI-level checks for edit/delete button visibility
